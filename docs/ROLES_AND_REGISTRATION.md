# Roles, Registration & Approval Flow

## Как это работает

1. Человек регистрируется через Clerk (`/login`). Самый первый пользователь
   системы автоматически становится **Owner** (кто-то должен одобрять остальных).
2. Все остальные после входа попадают на **`/onboarding`**: выбирают роль —
   Client, Dispatcher, Recruiter, Driver или Owner — заполняют имя/телефон,
   для Client/Driver указывают компанию, и отправляют запрос.
3. Все Owner/Admin мгновенно получают **уведомление** (колокольчик в Topbar +
   realtime-событие `user.registration.requested` + запись Notification).
4. Owner в разделе **Team** видит блок «Pending registrations»: может выдать
   запрошенную (или другую) роль, для Client — привязать к компании, для
   Driver — к профилю водителя, либо отклонить с причиной.
5. Пользователь в статусе ожидания видит экран «Waiting for approval»
   (автоматически обновляется); после одобрения его сразу пускает в его
   кабинет, при отказе — показывается причина и можно подать запрос заново.

Все действия пишутся в AuditLog (`role_requested` / `role_approved` /
`role_rejected`).

## Роли и доступы

| Роль | Доступ |
|---|---|
| **Owner** | Полный контроль: любые статусы, ручные исключения, справочник статусов, история, автоматизация, финансы, одобрение регистраций, ставки водителей |
| **Admin** | То же, что Owner (legacy-роль) |
| **Senior Dispatcher** | Команда: клиенты/водители/траки/грузы своей команды, карта, override статусов. Без Settings/Security |
| **Dispatcher** | Свои клиенты/водители/грузы, операционные статусы, карта. **Нет доступа к Reports и Security** |
| **Updater** | Обновление локаций/ETA и только операционные статусы пайплайна; без финансов и ручных исключений |
| **Recruiter** | Водители/компании/траки в рамках онбординга; не видит грузы, карту, финансы |
| **Finance** | Финансы, инвойсы, отчеты; без карты |
| **Client** | **Только просмотр**, только своя компания: свои водители, свои траки, грузы своих траков/водителей, карта своих водителей. Ничего изменить не может. Живет в `/portal` |
| **Driver** | Личный кабинет `/driver-app`: текущий груз, статусы, маршрут, документы и **My Finance** — мили, средний $/миля, gross, недельная динамика, персональный заработок (`payPerMile × мили`, ставку задает Owner в карточке водителя), утилизация времени по статусам, on-time % |

Роли Client и Driver жестко изолированы: CLIENT-аккаунт связан с `Client.id`
(`User.clientId`), DRIVER-аккаунт — с `Driver.id` (`User.driverId`); все их
запросы и realtime-события фильтруются по этой привязке на сервере.

## API

```
GET/POST /api/onboarding            состояние регистрации / запрос роли (работает для PENDING)
GET      /api/team/pending          ожидающие одобрения + списки для привязки (Owner)
POST     /api/team/:id/approve      выдать роль (+clientId/driverId)  (Owner)
POST     /api/team/:id/reject       отклонить с причиной              (Owner)
GET      /api/driver-app/finance    личные финансы водителя
PATCH    /api/drivers/:id           payPerMile (Owner)
```

## Применение схемы

```bash
npx prisma db push && npx prisma generate
```

SQL-эквивалент: `prisma/migrations/20260724_roles_and_registration/migration.sql`.
Существующие пользователи не затрагиваются (их статус остается ACTIVE).
